# CLAUDE.md 生成（CLAUDE.md Generation）

## Overview

設計フェーズで積み上げた各ドキュメント（要件・実現可能性・思想・Git運用・技術選定・設計・DB）を、
実装フェーズに入る前に **プロジェクトルートの `CLAUDE.md`** へ集約する。CLAUDE.md は Claude Code が
セッション開始時に読み込むプロジェクトメモリであり、ここに「どのドキュメントが何を定めているか」と
「実装・修正の進め方のルール」を書くことで、以降の実装が常にドキュメントに沿うようにする。

## When to Use

- 設計フェーズ（`/design`）が完了し、実装（`/implement`）に入る前
- 設計ドキュメントが追加・更新され、CLAUDE.md の記述が古くなったとき

**前提**: 設計成果物（少なくとも `docs/REQUIREMENTS.md` と `docs/DESIGN.md`）が存在すること。

## Process

### 1. 生成された設計ドキュメントを確認する

`docs/` 配下に存在するものを把握する（プロジェクトにより一部は無いこともある）:

- `docs/REQUIREMENTS.md` — 要件定義
- `docs/FEASIBILITY.md` — 実現可能性調査
- `docs/philosophy/PLAN_PHILOSOPHY.md` / `CODING_PHILOSOPHY.md` / `TEST_PHILOSOPHY.md` — 3思想
- `docs/GIT_CONVENTIONS.md` — Git運用方針
- `docs/TECH_STACK.md` — 技術選定
- `docs/DESIGN.md` — アーキ設計
- `docs/DATABASE.md` — DB設計（あれば）

### 2. 各ドキュメントの役割を1行で要約する

各ファイルの冒頭を読み、「何を定めているか」「いつ参照するか」を短くまとめる（存在するものだけ載せる）。

### 3. CLAUDE.md を生成する（既存があれば更新）

プロジェクトルートに `CLAUDE.md` を作成する。**既に存在する場合は上書きせず**、ドキュメント体系や
ルールの節だけを更新し、人が手で加えた記述は保持する（マージする）。下記テンプレートを用いる。

必ず含める要素（ユーザー要件）:
- **各ドキュメントの説明**（ドキュメント体系の表）
- **実装はドキュメントに沿って行う**という原則
- **修正時はコードとドキュメントを一緒に修正し整合を保つ**という運用ルール

### 4. CLAUDE.md テンプレート

````markdown
# <プロジェクト名> — Claude Code プロジェクトメモリ

このファイルは Claude Code がセッション開始時に読み込むプロジェクトの基準です。
作業の前に関連ドキュメントを必ず参照し、本ファイルのルールに従ってください。

## ドキュメント体系

| ドキュメント | 役割 | 参照タイミング |
|---|---|---|
| `docs/REQUIREMENTS.md` | 機能/非機能要件・受け入れ条件・スコープ | 「何を作るか/完了の定義」を確認するとき |
| `docs/FEASIBILITY.md` | 技術的実現可能性・リスク・PoC要否 | 技術的な制約・前提を確認するとき |
| `docs/philosophy/PLAN_PHILOSOPHY.md` | 設計思想（アーキ・構造の判断基準） | 構造・依存・分割を判断するとき |
| `docs/philosophy/CODING_PHILOSOPHY.md` | 実装思想（コードの書き方の基準） | コードを書く前に必ず |
| `docs/philosophy/TEST_PHILOSOPHY.md` | テスト思想（品質担保の基準） | テストを設計・実装するとき |
| `docs/GIT_CONVENTIONS.md` | Git運用方針（粒度・メッセージ規約） | コミット・ブランチ運用のとき |
| `docs/TECH_STACK.md` | 採用技術と選定理由 | 技術選択・依存追加のとき |
| `docs/DESIGN.md` | アーキ・コンポーネント・API/IF | 実装の全体像を確認するとき |
| `docs/DATABASE.md` | DBスキーマ・ER図（あれば） | データ層・マイグレーションのとき |

> 上表のうち、実在するドキュメントのみ記載する。

## 実装の原則（ドキュメント駆動）

- 実装に着手する前に、関連ドキュメント（最低でも `DESIGN.md` と `CODING_PHILOSOPHY.md`、
  データを扱うなら `DATABASE.md`）を読む。
- **コードはドキュメントに沿って実装する。** ドキュメントに反する独自判断をしない。
- ドキュメントとコードが食い違っている場合は、勝手に進めずどちらが正かを確認する
  （原則ドキュメントを正とし、設計変更なら先にドキュメントを直す）。

## 変更時のルール（コードとドキュメントを一緒に更新）

- **コードを変更したら、影響する設計ドキュメントも同じ変更で必ず更新する**（コードとドキュメントの整合を保つ）。
  - 例: API を変えたら `DESIGN.md`、スキーマを変えたら `DATABASE.md`、技術を差し替えたら `TECH_STACK.md`、
    要件が変わったら `REQUIREMENTS.md` を更新する。
- 仕様・設計の変更は、**先にドキュメントを更新して合意してから**コードに反映する。
- 思想（philosophy/*）や規約（GIT_CONVENTIONS）から逸脱する必要が出たら、勝手に破らず該当ドキュメントを
  更新する形で合意する。

## Git運用

- `docs/GIT_CONVENTIONS.md` のコミット粒度・メッセージ規約に従う。
- シークレット・APIキー・`.env` はコミットしない。

## テスト

- `docs/TEST_PHILOSOPHY.md` の比率・方針に従う。受け入れ条件（`REQUIREMENTS.md`）に対応するテストを用意する。

## 言語・その他

- ユーザーへの応答・説明は日本語で行う。
- エラーを握りつぶさない。変更したら最後に変更箇所を説明する。
````

### 5. 人間のレビューを受ける

生成・更新した `CLAUDE.md` を確認してもらい、プロジェクト固有の追記があれば反映する。

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「ドキュメントがあるなら CLAUDE.md は不要」 | 散在するドキュメントは読まれない。CLAUDE.md は入口であり、参照を強制する索引になる。 |
| 「実装ルールは暗黙で分かる」 | 暗黙だと毎回ぶれる。「ドキュメントに沿う/一緒に直す」を明文化して初めて守られる。 |
| 「修正時のドキュメント更新は後でまとめて」 | 後回しはほぼ放置され、ドキュメントとコードが乖離する。変更と同時に更新する。 |
| 「既存の CLAUDE.md を上書きでいい」 | 人が加えた記述を失う。索引とルールの節だけ更新し、手書き部分は保持する。 |
| 「無いドキュメントも一応載せる」 | 実在しない参照は混乱の元。存在するものだけ載せる。 |

## Red Flags

- 「実装はドキュメントに沿う」「修正時はドキュメントも更新」のルールが書かれていない
- 各ドキュメントの説明（役割・参照タイミング）が無い、または実在しないファイルを列挙している
- 既存の CLAUDE.md を確認せず上書きしている
- プロジェクトルートではない場所に置いている（Claude Code が読むのはルートの `CLAUDE.md`）

## Verification

実装フェーズに進む前に確認する:

- [ ] プロジェクトルートに `CLAUDE.md` が作成/更新された
- [ ] 実在する各設計ドキュメントの説明（役割・参照タイミング）が記載されている
- [ ] 「実装はドキュメントに沿って行う」原則が明記されている
- [ ] 「修正時はコードとドキュメントを一緒に更新する」ルールが明記されている
- [ ] Git運用・テスト・言語などの基本ルールへの参照がある
- [ ] 既存の手書き記述を失っていない（更新の場合）
- [ ] 人間がレビューした

## 次のフェーズ

CLAUDE.md を生成したら実装フェーズへ。`/implement`（`implementation` スキル）は CLAUDE.md を入口に
各ドキュメントを参照し、ドキュメントに沿って実装・修正する。
