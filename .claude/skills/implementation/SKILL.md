# 実装（Implementation）

## Overview

`docs/DESIGN.md`（何をどう作るか）と `docs/philosophy/CODING_PHILOSOPHY.md`（コードをどう書くか）に
準拠してコードを書く。実装は設計と実装思想の忠実な具現化であり、独断で方針を変えない。大きく一気に
作らず、動く垂直スライス単位で小さく積み上げる。

## When to Use

- 設計（DESIGN.md）が承認され、コードを書き始めるとき
- 既存機能を設計に沿って改修するとき

**前提**: `docs/philosophy/CODING_PHILOSOPHY.md` と `docs/DESIGN.md` が存在すること。
なければ先に `philosophy-definition` / `design-and-architecture` を実行。

## Process

### 1. 着手前に思想と設計を読む（必須）

- `docs/philosophy/CODING_PHILOSOPHY.md` — 命名・粒度・型・コメント・DRY・依存方針など
- `docs/DESIGN.md` — 実装対象のコンポーネント・データモデル・API/IF
- `docs/DATABASE.md`（あれば）— データ層の実装・マイグレーションの基準となるスキーマ・ER図
- `docs/REQUIREMENTS.md` の受け入れ条件 — 「完了」の定義
- 既存コード — 再利用できる関数・ユーティリティ・パターンを探す（新規作成より再利用優先）

### 2. 垂直スライスに分割する

横（レイヤー）ではなく縦（1つの完結した機能パス）で割る。各スライスは単体で動作確認・コミット可能にする。

```
良い分割: 「ログインAPI 1本（入力→検証→DB→応答）を端から端まで」
悪い分割: 「全モデル定義」→「全コントローラ」→「全UI」（横割りで動かせない）
```

### 3. スライスごとに実装する

各スライスで以下を回す:

1. CODING_PHILOSOPHY に沿って実装する
2. ローカルで動作確認する
3. **思想準拠を自己チェック**（下記チェックリスト）
4. 小さくコミットする（1コミット1論理変更、メッセージは「なぜ」を書く）

### 4. 思想準拠の自己チェック（各スライス）

CODING_PHILOSOPHY の決定に照らして確認する。代表例:
- [ ] 命名規則・関数粒度が思想どおり
- [ ] 型の厳密さが守られている（例: any を使っていない）
- [ ] 副作用が境界に寄っている／純粋関数志向が守られている
- [ ] エラーを握りつぶしていない（CLAUDE.md準拠）
- [ ] APIキー・シークレットを直書きしていない、`.env` を触っていない
- [ ] コメントは「なぜ」を書き、自明なものは付けていない
- [ ] 受け入れ条件のうち、このスライスが満たすものを満たしている

### 5. 逸脱時は停止して相談する

思想・設計どおりに実装できない事情が出たら、**黙って自己流に変えず停止し、ユーザーに選択肢を示して相談**する。
合意できたら DESIGN.md / CODING_PHILOSOPHY.md 側を更新してから進む。

### 6. 変更点を説明する

作業の最後に、どこを・なぜ変更したかを日本語で説明する（CLAUDE.md準拠）。

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「思想は読まなくても良いコードは書ける」 | 「良い」の基準は人とプロジェクトで違う。思想はその基準そのもの。読まないとレビューで揉める。 |
| 「まとめて一気に作る方が速い」 | 大きな塊はレビュー・デバッグ・巻き戻しが困難。垂直スライスで小さく出す方が結局速い。 |
| 「設計と違うけどこっちの方が良い」 | 良い案なら設計を更新してから実装する。黙って変えると設計と実装が乖離する。 |
| 「エラーはとりあえず握りつぶす」 | 握りつぶしは原因究明を不能にする。CLAUDE.md でも禁止。握らず適切に伝播・記録する。 |
| 「コミットは最後にまとめて」 | 巨大コミットはレビュー不能・revert不能。各スライスでコミットする。 |
| 「型エラーはanyで黙らせる」 | any は型安全を捨てる。CODING_PHILOSOPHY の厳密さに反する。unknown+絞り込みで対処。 |

## Red Flags

- CODING_PHILOSOPHY / DESIGN を読まずにコードを書き始めている
- 大量の未コミット変更が溜まっている
- `any` の乱用、エラーの握りつぶし、シークレットの直書き
- 設計と異なる実装を相談なく進めている
- 受け入れ条件と対応の取れない「ついで実装」が混ざっている
- 既存の再利用可能なユーティリティを無視して重複実装している

## Verification

テストフェーズに進む前に確認する:

- [ ] 着手前に CODING_PHILOSOPHY.md と DESIGN.md を読んだ
- [ ] 垂直スライス単位で実装し、各スライスをコミットした
- [ ] 各スライスで思想準拠の自己チェックを通過した
- [ ] エラーを握りつぶしていない／シークレットを直書きしていない
- [ ] 対象の受け入れ条件を満たしている
- [ ] 思想・設計からの逸脱は相談・合意し、ドキュメントへ反映した
- [ ] 変更点を日本語で説明した
