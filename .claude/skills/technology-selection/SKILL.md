# 技術選定（Technology Selection）

## Overview

アーキ設計に入る前に、採用する技術スタックを根拠付きで確定し `docs/TECH_STACK.md` に残す。
「なぜこの技術を選んだのか」「類似技術と比べてなぜこれなのか」を明文化することで、流行や好みではなく
要件・思想に紐づいた意思決定にし、後から決定の背景を辿れるようにする。アーキ設計は選定スタックを前提に
進むため、技術はここで確定させる。

## When to Use

- 設計フェーズで思想定義（PLAN_PHILOSOPHY）が済み、アーキ設計に入る前
- 言語・フレームワーク・DB・インフラなど技術選択が必要なとき
- 既存スタックに新技術を追加・置換する判断が必要なとき

**前提**: `docs/REQUIREMENTS.md` が存在すること。`docs/philosophy/PLAN_PHILOSOPHY.md` と
`docs/FEASIBILITY.md` があれば必ず入力にする（優先度と技術候補・制約を受け継ぐ）。

**使わない場面**: 技術が既に確定・固定されていて選択の余地がないとき（その場合は確定事項を TECH_STACK.md に
記録するだけでよい）。

## Process

### 1. 入力を読む

- `docs/REQUIREMENTS.md` — 機能/非機能要件（性能・スケール・対応環境・コスト制約）
- `docs/philosophy/PLAN_PHILOSOPHY.md` — 性能/拡張性/保守性などの優先度（評価軸の重み付けに使う）
- `docs/FEASIBILITY.md` — 実現可能性調査で挙がった技術候補・推奨・制約（出発点にする）
- 既存コード・技術スタック・チームの習熟（再利用・一貫性を優先）

### 2. 選定領域を洗い出す

プロジェクト規模に応じて、決めるべき領域を列挙する。例:
言語 / フロントエンドFW / バックエンドFW / DB（RDB/NoSQL）/ 状態管理 / 認証・認可 /
ホスティング・インフラ / ビルド・パッケージ管理 / テスト・CIツール / 監視。

### 3. 各領域で候補（類似技術スタック）を比較する

- 各領域につき**複数の候補**を挙げる（例: FE = React / Vue / Svelte）
- 評価軸で比較する: 要件適合 / PLAN_PHILOSOPHY の優先度との整合 / エコシステム成熟度・将来性 /
  学習コスト・チーム習熟 / 保守性 / 性能 / コスト・ライセンス / ロックイン・移行容易性
- **最新状況（バージョン・サポート状況・既知の問題）は WebSearch/WebFetch で裏取りし、出典を残す**

### 4. 採用を決定し、理由を明文化する

各領域で採用技術を決め、次の3点を必ず書く（ユーザー要件）。

1. **選定した技術**
2. **選定理由**（どの要件・優先度を満たすから選んだか）
3. **類似技術スタックと比べてなぜそれを選定したか**（比較した候補と、他を選ばなかった理由）

### 5. スタック全体の整合とリスクを確認する

- 採用技術同士の相性・一貫性（例: FW と DB クライアント、ホスティングの組み合わせ）
- ロックイン・移行リスク、採用バージョンと EOL、コストの見積もり

### 6. docs/TECH_STACK.md に保存し、人間のレビューを受ける

```markdown
# 技術スタック選定: <プロジェクト/機能名>

## 1. 概要・選定方針（PLAN_PHILOSOPHY の優先度との対応）
## 2. 選定サマリ（領域 × 採用技術 × 主な理由）
## 3. 領域別の選定詳細
   ### <領域名（例: フロントエンドフレームワーク）>
   - 採用技術: <技術名・バージョン>
   - 選定理由: <要件/優先度に基づく理由>
   - 比較した類似技術（候補比較表）:
     | 候補 | 要件適合 | 成熟度 | 学習/習熟 | 性能 | ロックイン | 採用可否 |
     |---|---|---|---|---|---|---|
   - なぜこれを選んだか / 他を選ばなかったか: <比較に基づく決め手と不採用理由>
   - 出典: <URL>
## 4. スタック全体の一貫性・相性
## 5. リスク・ロックイン・前提（バージョン・EOL・コスト）
## 6. 要件・思想との対応
```

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「いつものスタックでいい」 | 慣れは一つの評価軸でしかない。要件・優先度に照らして妥当か比較してから決める。 |
| 「人気だから/最新だから採用」 | 人気・新しさは保守性や安定性を保証しない。要件適合とエコシステム成熟度で判断する。 |
| 「比較は省いて結論だけ書く」 | 比較がないと後から再検討できない。類似技術との比較と不採用理由を残すのが本スキルの目的。 |
| 「思想と関係なく技術は選べる」 | 性能/拡張性/保守性の優先度（PLAN_PHILOSOPHY）が評価軸の重みを決める。思想に紐づける。 |
| 「バージョンは後で確認」 | EOL・既知の脆弱性・破壊的変更は選定を左右する。最新状況を裏取りしてから決める。 |
| 「ロックインは気にしない」 | 移行困難な選択は将来の足かせになる。ロックインを評価軸に入れて明示する。 |

## Red Flags

- 採用技術は書いてあるが「類似技術との比較」「不採用理由」がない
- 比較に評価軸がなく、好み・流行で選んでいる
- PLAN_PHILOSOPHY の優先度・要件と選定理由が結びついていない
- バージョン・サポート状況・ライセンスを確認していない
- 採用技術同士の相性・一貫性を検討していない
- 既存スタック・チーム習熟を無視して新技術を持ち込んでいる

## Verification

アーキ設計に進む前に確認する:

- [ ] 主要な選定領域すべてに採用技術が決まっている
- [ ] 各採用に「選定理由」と「類似技術との比較・不採用理由」がある
- [ ] 比較に評価軸（要件適合・成熟度・性能・ロックイン等）が使われている
- [ ] 選定理由が REQUIREMENTS と PLAN_PHILOSOPHY に紐づいている
- [ ] 最新状況（バージョン等）を裏取りし、出典を残した
- [ ] スタック全体の一貫性・リスク・ロックインを確認した
- [ ] 人間がレビューした
- [ ] `docs/TECH_STACK.md` に保存されている

## 次のステップ

技術が確定したら `design-and-architecture` スキルへ。`TECH_STACK.md` で採用したスタックを前提に、
アーキ概要・コンポーネント・データモデル・API/IF を設計する（`docs/DESIGN.md`）。
