# レビュー・監査（Review & Audit）

## Overview

テストが通った変更を、マージ／リリース前に複数の観点でレビュー・監査する。実装者本人とは別の視点で
「正しさ・可読性・設計・セキュリティ・性能・思想準拠」を点検し、見逃しを潰す最後のゲート。
専門ペルソナ（サブエージェント）に並行で委譲し、結果を統合して `docs/REVIEW.md` にまとめる。

本ワークフローでは **1機能=1ブランチ=1PR**。レビュー対象は当該機能ブランチの差分で、判定が
**マージ可**になったら AI が PR を提出し（push → `gh pr create`）、**マージは必ず人間が行う**
（AIは `main` にマージしない）。これにより `main` を常にグリーンに保つ。

## When to Use

- テストフェーズ（`/5-test`）が完了し、マージ／リリースの前
- プルリクエストのレビューを行うとき
- 大きな変更・セキュリティに関わる変更を入れる前

**使わない場面**: ドキュメントのみの変更、設定値の軽微な修正（その場合は該当観点だけ簡易確認）。

## Process

### 1. レビュー対象と入力を確定する

- レビュー対象の差分 = 当該機能ブランチ（`git diff main...feature/<ID>`）。コミット範囲や PR でも可
- `docs/REQUIREMENTS.md`（受け入れ条件 = 正しさの基準）
- `docs/TASKS.md`（当該タスクの範囲・受け入れ条件の紐づけ）
- `docs/DESIGN.md`（設計どおりか）
- `docs/philosophy/*.md`（思想準拠の基準）

### 2. 専門ペルソナに委譲する（並行実行）

`Agent` ツールで以下のサブエージェントを起動し、各観点を独立して点検させる。
独立した観点なので**1メッセージで並行起動**してよい。

| ペルソナ | 観点 |
|---|---|
| `code-reviewer` | 正しさ・可読性・設計・保守性（5観点の総合レビュー） |
| `security-auditor` | 認証/認可・入力検証・機微データ・OWASP 等のセキュリティ監査 |
| `web-performance-auditor` | Core Web Vitals・バンドル・レンダリング・N+1 等の性能監査 |
| `philosophy-compliance-reviewer` | `docs/philosophy/*.md` への準拠（本ワークフロー固有の観点） |

各ペルソナには対象差分と関連ドキュメントのパスを渡す。

### 3. 指摘を統合し、重大度で分類する

各ペルソナの指摘を集約し、重複を除き、重大度でラベリングする。

- **Blocker（必須）**: マージ前に必ず直す（バグ、セキュリティ欠陥、受け入れ条件違反、思想の重大逸脱）
- **Should（推奨）**: 直すべき（設計の歪み、可読性、性能リスク）
- **Consider（任意）**: 検討の余地（改善提案、好みの範囲）

各指摘には「ファイル:行 / 何が問題か / なぜ問題か / 推奨対応」を付ける。

### 4. docs/REVIEW.md にまとめ、判定する

```markdown
# レビュー・監査レポート: <対象>

## サマリ（判定: マージ可 / 要修正）
## Blocker（必須対応）
## Should（推奨対応）
## Consider（任意）
## 観点別メモ（コード / セキュリティ / 性能 / 思想準拠）
## 受け入れ条件の充足状況
```

### 5. 是正をループする

Blocker が残る間はマージ判定を「要修正」とし、修正 → 再レビューを回す。
修正は通常 `implementation`／`testing` スキルに戻って行い、思想・設計ドキュメントとの整合も保つ。

### 6. PRを提出する（マージ可のときのみ）

判定が**マージ可**になったら、当該機能ブランチを push し、`main` を base に PR を作成する。

```
git push -u origin feature/<ID>
gh pr create --base main --head feature/<ID> --title "<タスクタイトル>" --body "<本文>"
```

PR本文には「タスク概要 / 対応する受け入れ条件 / レビュー結果サマリ（Blocker 0 を含む） / テスト結果」を書く。
GitHubリモート/gh CLI が無い場合は、ブランチ名・PRタイトル・本文・実行コマンドを提示して人間に委ねる。
**要修正のままPRを出さない。**

### 7. 人間の承認・マージを待つ（AIはマージしない）

PRのマージは**必ず人間が行う**。AIは `git merge` やPRマージを実行しない。人間がマージしたら `main` を
最新化し、`docs/TASKS.md` の当該タスクを「マージ済み」に更新して、次タスクの実装（`/4-implement`）へ戻る。

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「テストが通ったからレビューは不要」 | テストは想定したことしか検証しない。レビューは想定外（設計の歪み・セキュリティ）を捕まえる。 |
| 「自分で書いたから自分で見れば十分」 | 著者は自分の思い込みを見抜けない。別視点（別ペルソナ）が見逃しを潰す。 |
| 「セキュリティは専門家がいないから飛ばす」 | security-auditor ペルソナが OWASP 観点で点検する。飛ばす理由にはならない。 |
| 「思想準拠まで見るのはやり過ぎ」 | 思想逸脱の放置はコードベースの一貫性を崩す。本ワークフローの肝なので必ず点検する。 |
| 「Consider まで対応すると終わらない」 | 重大度分類の意味はそこ。Blocker を必須、Consider は任意として割り切る。 |

## Red Flags

- 受け入れ条件との照合をせずに「LGTM」している
- セキュリティ・性能・思想準拠のいずれかの観点が完全に抜けている
- 指摘に「なぜ問題か」「推奨対応」が書かれていない
- Blocker が残っているのにマージ可と判定している／要修正のままPRを出している
- AI が `main` にマージしている（マージは必ず人間が行う）
- レビューを著者一人だけで完結させている（別ペルソナの視点がない）

## Verification

マージ／リリースを判定する前に確認する:

- [ ] 4ペルソナ（コード/セキュリティ/性能/思想準拠）の観点を点検した
- [ ] 各指摘に 重大度・場所・理由・推奨対応 が付いている
- [ ] 受け入れ条件の充足状況を確認した
- [ ] Blocker が0、または残課題が明示・合意されている
- [ ] `docs/REVIEW.md` に保存されている
- [ ] 判定（マージ可 / 要修正）が明記されている
- [ ] マージ可なら PR を提出した（push → `gh pr create`）。AIは `main` にマージしていない
- [ ] マージは人間に委ね、マージ後に `docs/TASKS.md` の状態を更新する流れになっている

## フェーズ完了

これで「要件定義 → 設計 → 実装 → テスト → レビュー・監査」の一巡が完了する。
是正が必要なら該当フェーズへ戻り、ドキュメント（要件・思想・設計）との整合を保ったまま修正する。
