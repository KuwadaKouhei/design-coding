# 設計・アーキテクチャ（Design & Architecture）

## Overview

`docs/REQUIREMENTS.md`（何を作るか）と `docs/philosophy/PLAN_PHILOSOPHY.md`（どう構造化するか）を入力に、
**具体的な設計**を `docs/DESIGN.md` として書き出す。設計は実装の地図であり、思想を個別の要件に適用した
結果である。思想に反する判断をするときは、その理由を明示する。

## When to Use

- 要件定義が完了し、設計思想（PLAN_PHILOSOPHY）が確定したあと
- 実装に着手する前
- アーキテクチャ上の意思決定が必要なとき

**前提**: `docs/philosophy/PLAN_PHILOSOPHY.md` が存在すること。なければ先に `philosophy-definition` を実行。

## Process

### 1. 入力を読む

- `docs/REQUIREMENTS.md` — 機能/非機能要件・受け入れ条件・スコープ
- `docs/philosophy/PLAN_PHILOSOPHY.md` — 設計思想（アーキスタイル・境界・依存方向・優先順位）
- `docs/FEASIBILITY.md` — 実現可能性調査で判明した制約・リスク・推奨技術・PoC要否（設計の前提にする）
- `docs/TECH_STACK.md` — 技術選定で確定した採用技術（このスタックを前提にアーキを設計する）
- 既存コードがあれば、再利用できる構造・パターン・ユーティリティを探す（新規作成より再利用を優先）

### 2. 設計思想を各要件に適用する

PLAN_PHILOSOPHY の決定を、今回の機能の具体に落とす。例:
- 「ドメイン単位で分割」→ 今回はどんなドメイン／モジュールに割るか
- 「ドメインを外部詳細に依存させない」→ DB・外部API をどこで隔離するか

### 3. 設計を組み立てる

以下を検討し決める。

- **アーキテクチャ概要**: 全体構成図（ASCII可）、レイヤー／モジュールの責務
- **コンポーネント分割**: 主要コンポーネント／モジュールと責務・依存関係
- **データモデル**: エンティティ、属性、関連、永続化方式
- **API / インターフェース設計**: エンドポイント or 関数シグネチャ、入出力、エラー応答
- **主要な処理フロー**: 重要なユースケースのシーケンス（リクエスト→応答）
- **横断的関心事**: 認証・認可、バリデーション、エラーハンドリング、ロギング
- **トレードオフと代替案**: 採用案と却下案、その理由

### 4. 思想との整合を確認し、逸脱を記録する

各設計判断が PLAN_PHILOSOPHY に沿っているか自己点検する。やむを得ず逸脱する場合は、
DESIGN.md に「逸脱と理由」を明記する（黙って逸脱しない）。

### 5. docs/DESIGN.md に保存し、人間のレビューを受ける

```markdown
# 設計: <機能名>

## 1. 概要 / 設計方針（PLAN_PHILOSOPHYとの対応）
## 2. アーキテクチャ概要（構成図）
## 3. コンポーネント / モジュール構成
## 4. データモデル
## 5. API / インターフェース設計
## 6. 主要処理フロー
## 7. 横断的関心事（認証・エラー・ログ等）
## 8. トレードオフ・代替案・却下理由
## 9. 設計思想からの逸脱（あれば、理由付き）
## 10. 未決事項
```

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「設計思想を読まなくても良い設計はできる」 | 思想を無視すると判断がぶれ、レビューで「なぜこうした？」が頻発する。思想は判断の近道。 |
| 「設計図は要らない、コードが設計だ」 | コードは詳細すぎて全体像が見えない。設計図は合意とレビューのための抽象。 |
| 「データモデルは作りながら決める」 | データモデルの誤りは最も修正コストが高い。先に固める価値が最大。 |
| 「トレードオフは書かなくても分かる」 | 数週間後の自分や他者には分からない。却下理由こそ将来の再検討を防ぐ。 |
| 「思想から外れるけど些細だから黙ってやる」 | 黙った逸脱は一貫性を壊す。明記すれば合意のうえの例外になる。 |

## Red Flags

- PLAN_PHILOSOPHY を読まずに設計を始めている
- データモデルや API のエラー応答が定義されていない
- 非機能要件（性能・セキュリティ）が設計に反映されていない
- トレードオフ・代替案の記述が一切ない
- 思想に反しているのに理由が書かれていない
- 設計レビューを飛ばして実装に進もうとしている

## Verification

実装フェーズに進む前に確認する:

- [ ] PLAN_PHILOSOPHY と REQUIREMENTS の両方を入力にしている
- [ ] アーキテクチャ概要・コンポーネント・データモデル・API/IF がそろっている
- [ ] 主要ユースケースの処理フローが書かれている
- [ ] エラーハンドリング・認証などの横断的関心事が設計されている
- [ ] トレードオフと却下理由が記録されている
- [ ] 思想からの逸脱があれば理由付きで明記されている
- [ ] 人間がレビューし承認した
- [ ] `docs/DESIGN.md` に保存されている

## 次のステップ

ここで定めるのは**概念データモデル**（主要エンティティと関連の概要）まで。要件達成に永続データが必要な
場合は、続けて `database-design` スキルで**物理スキーマとER図**（`docs/DATABASE.md`）を設計する。
永続データが不要なら DB設計はスキップする。

その後、設計完了後に `/3-claude-md` で CLAUDE.md を生成し、`/4-implement` コマンド（`implementation` スキル）で
実装へ進む。実装は `docs/DESIGN.md`・`docs/DATABASE.md`（あれば）・`docs/philosophy/CODING_PHILOSOPHY.md` に
準拠して進める。
