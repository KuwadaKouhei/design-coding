# 実現可能性調査（Feasibility Study）

## Overview

アーキテクチャを決める前に「そもそも作れるのか」「制約・リスクは何か」を技術的に検証する。
要件を鵜呑みにせず、必要技術の有無・成熟度・制約・性能限界を**裏取り**し、実現可能性を
`docs/FEASIBILITY.md` にまとめる。実現性が不確かなまま思想・アーキを固めると後工程で手戻りが起きる。
このスキルは設計フェーズの入口に立つゲートであり、楽観バイアスを排して不確実性を表に出すのが役割。

## When to Use

- 設計フェーズの最初（思想定義・アーキ設計の前）
- 新しい技術・外部API・未知のプラットフォーム制約が要件に絡むとき
- 性能・スケール・コンプライアンス上、実現性が読めない要件があるとき

**使わない場面**: 既知技術だけで作れることが自明な軽微変更（その場合は該当リスクだけ一言確認）。

## Process

### 1. 入力を読み、検証単位に分解する

- `docs/REQUIREMENTS.md` の機能/非機能要件・受け入れ条件・前提・制約を読む
- 既存コード・依存・インフラを把握する（再利用できる仕組みがあるか）
- 各要件を「技術的に検証可能な問い」に分解する（例: 「リアルタイム通知」→ WebSocket/SSE は対応環境で使えるか）

### 2. 各要件の実現可能性を調査する（裏取り必須）

次の観点で調べる。**推測で判断せず、社内コードと外部一次情報で裏を取る。**

- 必要技術・API・ライブラリの有無と成熟度（メンテ状況・既知の制限）
- プラットフォーム / ブラウザ / デバイスの対応状況
- 性能・スケール上の限界（想定データ量・同時接続・レイテンシ）
- 既存コード・インフラ・技術スタックとの整合
- ライセンス・コスト・コンプライアンス上の制約

調査手段:
- 社内: `Read` / `Grep` / `Glob` で既存実装・依存を確認、`Bash` でバージョンや動作を確認
- 外部: `WebSearch` / `WebFetch` で公式ドキュメント・仕様・既知の制限を確認し、**出典（URL）を控える**

### 3. リスク・不確実性を洗い出し、PoC要否を特定する

- 何が分かっていないか（不確実性）を明示する
- 「やってみないと分からない」箇所は **PoC / スパイク** が必要と特定し、検証方法を書く

### 4. 代替技術・代替アプローチを比較する

主要な技術選定について、候補・トレードオフ・推奨を整理する（最終決定はアーキ設計で行うが、土台を示す）。

### 5. 判定する

各要件を次の3段階で判定し、**根拠（できれば出典）** を必ず添える。

- **実現可能（Feasible）**: 既知の手段で問題なく作れる
- **条件付き可能（Feasible with conditions）**: 制約・前提・追加検証を満たせば可能
- **困難・要再検討（Hard / Re-scope）**: 現状の手段では難しい。要件の見直しが必要

### 6. docs/FEASIBILITY.md に保存し、レビュー・差し戻す

「困難」判定があれば、要件定義フェーズ（`/1-requirements`）への差し戻しを提案する。
以下の構成で保存する。

```markdown
# 実現可能性調査: <プロジェクト/機能名>

## 1. 概要・調査範囲
## 2. 判定サマリ（要件 × 判定 × 主なリスク）
## 3. 要件別の調査詳細（観点・根拠・出典）
## 4. 主要リスクと不確実性
## 5. PoC・検証が必要な項目（検証方法つき）
## 6. 技術選定の比較と推奨
## 7. 前提・制約（性能予算・対応環境・ライセンス等）
## 8. 要件への差し戻し提案（困難判定がある場合）
```

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「経験上できるはずだから調査不要」 | 経験は古くなる。API の仕様変更・廃止・制限は調べないと分からない。裏を取る。 |
| 「作りながら確かめればいい」 | 実現困難が実装中に発覚すると手戻りが最大化する。設計前に潰すのが最も安い。 |
| 「動くライブラリが見つかったからOK」 | メンテ状況・ライセンス・性能・制限を見ないと地雷を踏む。成熟度まで確認する。 |
| 「リスクは書くと不安を煽る」 | 隠したリスクは必ず後で爆発する。明示して対策・PoC につなげるのが役割。 |
| 「非機能（性能・スケール）は後で」 | 性能限界はアーキを左右する。実現可能性の中核なので先に見積もる。 |
| 「出典まではいらない」 | 出典のない判定は再検証できない。判断の根拠（URL）を残す。 |

## Red Flags

- 判定の根拠・出典がなく「たぶん大丈夫」で済ませている
- 非機能要件（性能・スケール・対応環境）の実現性に触れていない
- 不確実性・PoC 要否が一つも挙がっていない（楽観バイアス）
- 採用予定ライブラリのライセンス・メンテ状況を確認していない
- 「困難」判定なのに要件への差し戻しを提案していない
- 外部技術を社内の思い込みだけで判断し、一次情報を確認していない

## Verification

思想定義・アーキ設計に進む前に確認する:

- [ ] すべての主要要件に実現可能性の判定（実現可能/条件付き/困難）が付いている
- [ ] 各判定に根拠があり、外部技術には出典がある
- [ ] 非機能要件（性能・スケール・対応環境）の実現性を評価した
- [ ] リスク・不確実性と PoC 要否が整理されている
- [ ] 主要な技術選定の比較と推奨がある
- [ ] 「困難」判定があれば要件への差し戻しを提案している
- [ ] 人間がレビューした
- [ ] `docs/FEASIBILITY.md` に保存されている

## 次のステップ

実現可能性が確認できたら、設計フェーズの続き（`philosophy-definition` → `design-and-architecture`）へ進む。
調査で判明した制約・リスク・推奨技術は、設計思想（特に性能/拡張性の優先度）とアーキ設計の前提になる。
「困難」判定が残る場合は `/1-requirements` に戻り、要件を見直す。
