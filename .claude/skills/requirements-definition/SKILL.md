# 要件定義（Requirements Definition）

## Overview

コードを書く前に「何を・なぜ・どこまで作るのか」を文書として確定させる。要件定義は人間のエンジニアと
AI の共通の真実（source of truth）であり、以降の設計・実装・テストすべての出発点になる。
要件のないコードは推測でしかない。

## When to Use

- 新規プロジェクト／新機能に着手するとき
- 要望が曖昧・断片的・口頭のアイデア段階のとき
- 変更が複数ファイル・複数モジュールにまたがるとき
- 「完了」の定義が共有されていないとき

**使わない場面**: タイポ修正、1行の自明な変更、要件が完全に自己完結している軽微な修正。

## Process

要件定義は次の順で進める。各ステップは前のステップが確定してから次へ進む。

### 1. コンテキストを掴む

- このプロダクト／機能は誰のどんな課題を解決するのか
- ビジネス上のゴールと成功指標は何か
- 既存システム・既存コードとの関係（新規 / 追加 / 改修）

### 2. 機能要件を洗い出す

- ユーザーが実行できる操作（動詞で列挙）
- 各操作の入力・出力・前提条件
- 主要な画面 / API / データの流れ

### 3. 非機能要件を確認する

- パフォーマンス（応答時間、想定同時接続、データ量）
- セキュリティ（認証・認可、機微データ、コンプライアンス）
- 可用性・信頼性、アクセシビリティ、対応ブラウザ／デバイス
- 運用・監視・ログの要否

### 4. ユーザーストーリーと受け入れ条件に落とす

曖昧な要望を**テスト可能な受け入れ条件**へ変換する。

```
要望:   「ログインできるようにしたい」
ストーリー: ユーザーとして、メールとパスワードでログインしたい。安全に自分のデータにアクセスするため。
受け入れ条件:
  - [ ] 正しい資格情報でログインするとダッシュボードに遷移する
  - [ ] 誤った資格情報では「メールまたはパスワードが違います」を表示する
  - [ ] 5回連続失敗で15分間アカウントをロックする
  - [ ] パスワードは平文で保存・ログ出力されない
```

### 5. スコープ境界を定義する

- **やること（In scope）**
- **今回やらないこと（Out of scope / Not doing）** ← スコープクリープ防止に必須
- 既知の前提・制約・依存（外部API、納期、技術スタック）
- 未決事項（Open questions）と確認先

### 6. docs/REQUIREMENTS.md に保存し、人間のレビューを受ける

以下の構成で保存する。

```markdown
# 要件定義: <プロジェクト/機能名>

## 1. 背景・目的
## 2. 対象ユーザー / ペルソナ
## 3. スコープ（やること / やらないこと）
## 4. 機能要件
## 5. 非機能要件
## 6. ユーザーストーリーと受け入れ条件
## 7. 前提・制約・依存
## 8. 未決事項（Open Questions）
```

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「単純だから要件定義は要らない」 | 単純なタスクに*長い*要件は不要だが、受け入れ条件は必要。2〜3行でもよい。 |
| 「作りながら決めればいい」 | 後から書くのは仕様ではなくただの記録。要件の価値はコード前に曖昧さを潰すこと。 |
| 「要件は要望を聞けば分かる」 | 明確に見える要望にも暗黙の前提がある。受け入れ条件にして初めて差異が見える。 |
| 「どうせ要件は変わる」 | だから要件定義は生きた文書として更新する。古い要件でも無い要件よりはるかにまし。 |
| 「非機能要件は後で考える」 | 性能・セキュリティは設計を左右する。後付けは作り直しを招く。 |

## Red Flags

- 受け入れ条件が「ちゃんと動く」「いい感じ」のような検証不能な表現
- 「やらないこと」が一つも書かれていない
- 非機能要件（性能・セキュリティ・アクセシビリティ）が完全に欠落している
- 要望をそのままコードに翻訳し始めている
- ユーザーがレビューしていないまま設計に進もうとしている

## Verification

設計フェーズに進む前に確認する:

- [ ] 機能要件・非機能要件の両方が記載されている
- [ ] すべての受け入れ条件がテスト可能（観測可能な結果で書かれている）
- [ ] スコープ外（Not doing）が明示されている
- [ ] 未決事項が列挙され、確認の段取りがある
- [ ] 人間がレビューし承認した
- [ ] `docs/REQUIREMENTS.md` に保存されている

## 次のフェーズ

要件が確定したら設計フェーズへ。`/design` コマンド（`design-and-architecture` スキル）を実行する。
設計フェーズの冒頭で `philosophy-definition` スキルにより設計・実装・テストの「思想」を確定する。
