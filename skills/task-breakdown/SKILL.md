# タスク分解（Task Breakdown）

## Overview

`docs/DESIGN.md`（何を作るか）と `docs/REQUIREMENTS.md`（受け入れ条件）を、実装フェーズが回せる
**機能単位のタスク**へ分解し、実装順序・依存・推奨ブランチ名を決めて `docs/TASKS.md` にまとめる。
この一覧が実装フェーズの単一の真実（駆動表）になり、**1タスク=1機能=1ブランチ=1PR** で進める。
各タスクは単体で `main` にマージしても全体が壊れない縦スライスとし、`main` を常にグリーンに保つ。

## When to Use

- アーキ設計（`docs/DESIGN.md`）が確定し、実装に入る前（設計フェーズの最後のサブステップ）
- 大きな機能追加で、作業をPR可能な単位へ分割したいとき

**前提**: `docs/DESIGN.md` と `docs/REQUIREMENTS.md` が存在すること。`docs/DATABASE.md`・
`docs/GIT_CONVENTIONS.md` があれば参照する。なければ先に `design-and-architecture` /
`requirements-definition` を実行。

## Process

### 1. 入力を読む

- `docs/DESIGN.md` — コンポーネント・データモデル・API/IF（分解対象）
- `docs/REQUIREMENTS.md` — 受け入れ条件（各タスクの紐づけ先・「完了」の定義）
- `docs/DATABASE.md`（あれば）— データ層の作成順序の手がかり
- `docs/DIRECTORY_STRUCTURE.md`（あれば）— 各タスクの成果物の配置先・命名規約の参考
- `docs/GIT_CONVENTIONS.md`（あれば）— ブランチ命名規約（`feature/<ID>-<slug>` 等）

### 2. 機能を縦スライスのタスクに割る

横（レイヤー）ではなく縦（1つの完結した機能パス）で割る。各タスクは単体で動作確認・PR・マージ可能にする。

```
良い分割: 「ログイン機能（入力→検証→DB→応答→UI）を端から端まで」= 1タスク
悪い分割: 「全モデル定義」「全API」「全UI」= 横割りで単体ではmainを動かせない
```

粒度の目安: 大きすぎる（複数機能を1タスクに詰める）／小さすぎる（PRに値しない雑務）を避ける。
基盤・認証など他タスクの土台になるものを先に置く。

### 3. 各タスクに属性を付ける

各タスクに以下を持たせる:

- **ID**: `T01`, `T02` … の連番
- **タイトル**: 機能名（簡潔に）
- **概要**: 何を実装するか（1〜3行）
- **対応する受け入れ条件**: `docs/REQUIREMENTS.md` のどの条件を満たすか
- **依存タスク**: 先に終わっている必要がある ID（なければ「なし」）
- **推奨ブランチ名**: `feature/<ID>-<slug>`（GIT_CONVENTIONS の命名規約に従う）
- **状態**: `⬜未着手` / `🚧作業中` / `🟡PR中` / `✅マージ済み`（初期は全て未着手）

### 4. 実装順に並べる

依存を満たす順（先に動く土台から）に並べ替える。各タスクをマージした時点で `main` がグリーンに
保てる順序にする。並行可能なタスクはその旨を残す。

### 5. docs/TASKS.md にまとめる

次の構成で出力する:

```markdown
# 実装タスク一覧（TASKS）

> 設計（DESIGN.md）を機能単位に分解した実装の駆動表。実装フェーズは上から順に
> 1タスク=1ブランチ=1PR で進め、各PRは人間がマージしてから次タスクへ移る（GIT_CONVENTIONS 準拠）。

## 進め方
- 次の未完了タスクを選ぶ → `main` を最新化 → 推奨ブランチ名でブランチを切る → 実装・テスト・レビュー
  → PR提出 → 人間がマージ → 次タスクへ。

## タスク一覧
| ID | タイトル | 概要 | 受け入れ条件 | 依存 | ブランチ名 | 状態 |
|----|---------|------|-------------|------|-----------|------|
| T01 | <機能> | <概要> | <REQ-x> | なし | `feature/T01-<slug>` | ⬜ |
| T02 | <機能> | <概要> | <REQ-y> | T01 | `feature/T02-<slug>` | ⬜ |

## 補足
- 並行可能: <あれば>
- リスク/未確定: <あれば>

## 更新履歴
- <YYYY-MM-DD> 初版作成
```

### 6. 人間のレビューを受ける

並び順・粒度・受け入れ条件の網羅をユーザーに確認する。タスクは「生きた文書」で、実装中に分割・
追加が必要になれば本ドキュメントを更新する（状態列も実装の進行に合わせて更新する）。

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「設計を読めば実装順は自明」 | 自明なら明文化のコストは低い。明文化しないとブランチ粒度がぶれ、PRが巨大化する。 |
| 「まとめて1ブランチで作る方が速い」 | 巨大PRはレビュー・マージ・revert が困難。1機能=1ブランチが結局速く安全。 |
| 「受け入れ条件との対応は後で」 | 後だと取りこぼす。分解時に紐づけておけば実装・レビューの基準が一意になる。 |
| 「依存はやりながら気付けばいい」 | 依存を無視した順序は `main` を壊す。先に並べておけば常時グリーンを保てる。 |

## Red Flags

- 横割り（「全モデル」「全API」）でタスクを切っている（単体で `main` を動かせない）
- どの受け入れ条件にも紐づかないタスクがある／条件を取りこぼしている
- 1タスクが複数機能を含み、PRが巨大になる粒度
- 依存関係・実装順が無く、並びがランダム
- ブランチ名が未定で、実装フェーズがブランチを切れない

## Verification

実装フェーズに進む前に確認する:

- [ ] 全機能がタスクに分解され、ID が採番されている
- [ ] 各タスクが受け入れ条件に紐づいている（取りこぼしがない）
- [ ] 各タスクに推奨ブランチ名（`feature/<ID>-<slug>`）が付いている
- [ ] 依存を満たす実装順に並んでいる（各マージ時点で `main` がグリーン）
- [ ] 粒度が 1機能=1ブランチ=1PR に収まっている
- [ ] `docs/TASKS.md` に保存され、人間のレビューを受けた

## 次のフェーズ

タスク分解が完了したら設計フェーズは終了。`/3-claude-md` で CLAUDE.md を生成し、`/4-implement` で
`docs/TASKS.md` を駆動表に、1タスク=1ブランチで実装に入る。
