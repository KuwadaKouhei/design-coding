# 思想定義（Philosophy Definition）

## Overview

設計に着手する前に、プロジェクトを貫く3つの「思想（哲学）」を言語化して固定する。

- **設計思想（PLAN_PHILOSOPHY）**: アーキテクチャ・構造をどう考えるか
- **実装思想（CODING_PHILOSOPHY）**: コードをどう書くか
- **テスト思想（TEST_PHILOSOPHY）**: 品質をどう担保するか

これらは個別の判断のたびに発生する「迷い」をあらかじめ解消する憲法であり、AI と人間が同じ価値観で
設計・実装・テストを進めるための基盤になる。**口頭で察するのではなく、インタビューで明示的に引き出し、
ドキュメントに残す**のがこのスキルの目的。

3思想を確定した直後に、続けて **Git運用方針**（①Git管理の有無 ②コミット粒度 ③コミットメッセージ規約）も
インタビューし、`docs/GIT_CONVENTIONS.md` にまとめる。実装フェーズのコミットはこの規約に準拠する。

## When to Use

- 設計フェーズの最初（`/2-design` の冒頭）
- `docs/philosophy/` の3ファイルがまだ存在しないとき
- 既存の思想を見直し・更新したいとき

**使わない場面**: 思想ドキュメントが既に存在し、今回の変更がその範囲内で完結するとき
（その場合は既存の `docs/philosophy/*.md` を読んで準拠するだけでよい）。

## Process

インタビュー形式で進める。**一度に全部を聞かない。1問ずつ、ユーザーの回答を受けてから次に進む。**

### 1. 既存ドキュメントを確認する

- `docs/philosophy/` に `PLAN_PHILOSOPHY.md` / `CODING_PHILOSOPHY.md` / `TEST_PHILOSOPHY.md` があるか確認
- `docs/REQUIREMENTS.md` があれば読み、要件から思想に効く制約（性能要件・チーム規模・技術スタック）を把握
- 既存ファイルがある場合は「更新するか・そのまま使うか」をユーザーに確認

### 2. 3つの思想を順にインタビューする

`question-bank.md`（このスキルディレクトリ内）の質問群を使う。進め方:

1. **設計思想 → 実装思想 → テスト思想** の順で扱う
2. 各テーマについて**1問ずつ**提示する。各問いには:
   - なぜこれを決める必要があるか（1行）
   - 代表的な選択肢と、プロジェクト状況から見た**推奨デフォルト**
   - を添える。`AskUserQuestion` ツールで選択肢提示すると回答しやすい
3. ユーザーの回答（と理由）をその都度メモする。回答が「お任せ」なら推奨デフォルトを採用し、その旨を記録
4. 重要な論点では「なぜそう考えるか」を一段深掘りする（インタビュー形式の肝）

> 全質問を機械的に消化しない。プロジェクト規模・要件に対して**重要な論点に時間をかけ**、
> 自明な論点はデフォルト採用で素早く流す。1思想あたり数問〜十数問が目安。

### 3. 回答を要約し、ユーザーの承認を得る

各思想について「決定事項リスト」を箇条書きで提示し、認識合わせをする。
ここで矛盾や抜けがあれば追加で質問して埋める。承認が取れてから次へ進む。

### 4. 3つのドキュメントを書き出す

`templates.md`（このスキルディレクトリ内）のテンプレートに沿って、`docs/philosophy/` 配下に3ファイルを
作成する。各ファイルには「原則 / 採用した決定 / 理由 / アンチパターン / 具体例」を含める。
**決定だけでなく理由（なぜそれを選んだか）を必ず残す** ——後から読む人が判断の背景を理解できるように。

### 5. Git運用方針をインタビューする（3思想の直後）

`question-bank.md` の **セクション D. Git運用方針** を使い、同じインタビュー形式で確定する。

1. まず **Git管理の有無** を聞く。使わない場合は理由を記録し、以降のGit質問をスキップする
2. 使う場合は **コミット粒度** と **コミットメッセージ規約**（と関連してブランチ運用）を1問ずつ深掘りする
3. 回答（と理由）を記録し、要約してユーザーの承認を得る

### 6. docs/GIT_CONVENTIONS.md を書き出す

`templates.md` の GIT_CONVENTIONS テンプレートに沿って `docs/GIT_CONVENTIONS.md` を作成する
（思想ファイルとは分けた独立ドキュメント）。シークレット・`.env` のコミット禁止を必ず明記する。
書き出し後、生成したファイルパス（思想3ファイル＋GIT_CONVENTIONS.md）をユーザーに報告する。

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「思想なんて大げさ、コードを書けばわかる」 | 思想が暗黙だと、判断のたびに方針がぶれる。明文化が一貫性とレビュー効率を生む。 |
| 「ベストプラクティスに従えば思想は不要」 | ベストプラクティスは文脈依存で衝突する（DRY vs シンプル等）。どちらを優先するかがまさに思想。 |
| 「全部AIにお任せでいい」 | お任せは可。ただし採用したデフォルトを文書化し、ユーザーが後から見直せる状態にする。 |
| 「一度決めたら変えられないのでは」 | 思想は生きた文書。要件変化に応じて更新してよい。固定ではなく現時点の合意。 |
| 「3つも作るのは面倒」 | 設計・実装・テストは関心が異なり混在すると参照しづらい。分離が後工程の参照を速くする。 |
| 「質問が多くて時間がかかる」 | 重要論点に絞ればよい。ここでの数十分が実装・レビューの手戻りを何時間も防ぐ。 |

## Red Flags

- ユーザーに一度も質問せず思想ドキュメントを書き始めている
- 「決定」だけ書いて「理由」が書かれていない
- 全質問を一括で投げて回答を待っている（インタビューになっていない）
- 要件（REQUIREMENTS.md）を読まずに思想を決めている
- 3つの思想に矛盾がある（例: 実装思想で「型を厳密に」なのにテスト思想で型を無視）
- ファイル名・フォルダ名が規約（docs/philosophy/、英大文字スネーク）と違う

## Verification

設計（アーキテクチャ）作業に進む前に確認する:

- [ ] 設計・実装・テストの3思想すべてをインタビューした
- [ ] 各決定に「理由」が付いている
- [ ] ユーザーが各思想の決定事項を承認した
- [ ] 3思想の間に矛盾がない
- [ ] `docs/philosophy/PLAN_PHILOSOPHY.md` が作成された
- [ ] `docs/philosophy/CODING_PHILOSOPHY.md` が作成された
- [ ] `docs/philosophy/TEST_PHILOSOPHY.md` が作成された
- [ ] Git運用方針（管理の有無・コミット粒度・メッセージ規約）をインタビューした
- [ ] `docs/GIT_CONVENTIONS.md` が作成された（Git不使用ならその判断と理由を記録した）

## 関連ファイル

- `question-bank.md` — 3思想＋Git運用方針のインタビュー質問集（このディレクトリ内）
- `templates.md` — 3思想ドキュメント＋GIT_CONVENTIONS.md の出力テンプレート（このディレクトリ内）

## 次のステップ

3思想とGit運用方針が確定したら `design-and-architecture` スキルへ。`PLAN_PHILOSOPHY.md` と
`REQUIREMENTS.md` に基づいて具体的なアーキテクチャ設計（`docs/DESIGN.md`）を行う。
`docs/GIT_CONVENTIONS.md` は実装フェーズのコミットの拠り所になる。
