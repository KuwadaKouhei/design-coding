# 思想インタビュー質問集（Question Bank）

`philosophy-definition` スキルの Process ステップ2で使う質問集。**1問ずつ**提示し、各問いには
「なぜ決めるか」「代表的な選択肢」「推奨デフォルト」を添える。すべてを機械的に聞かず、プロジェクト
規模・要件に応じて重要な論点を優先する。各回答は理由とともに記録する。

> 推奨デフォルトは「中規模の一般的な Web アプリ／少人数チーム」を想定した値。要件に応じて調整する。

---

## A. 設計思想（PLAN_PHILOSOPHY）

アーキテクチャ・構造をどう考えるか。

1. **アーキテクチャスタイル** — 全体構造の型は？
   - 選択肢: レイヤード / ヘキサゴナル（ポート&アダプタ）/ クリーンアーキテクチャ / 機能単位（feature-based）/ MVC / シンプルな単層
   - 推奨デフォルト: 機能単位（feature-based）＋必要に応じ軽いレイヤー分離
   - 深掘り: なぜそのスタイル？将来の規模拡大をどこまで見込むか？

2. **モジュール／コンポーネントの境界** — 何を基準に分割するか？
   - 選択肢: ドメイン（業務概念）単位 / 技術レイヤー単位 / 画面・ルート単位
   - 推奨デフォルト: ドメイン単位で分割し、その中で技術レイヤーを薄く持つ

3. **依存関係の方向** — 依存はどちらへ向けるか？
   - 選択肢: 内側（ドメイン）へ向ける（依存性逆転）/ 上から下への素直な依存 / 特に制約なし
   - 推奨デフォルト: ドメイン（ビジネスロジック）を外部詳細（DB・UI・API）に依存させない

4. **状態管理** — アプリ状態をどこで・どう持つか？（フロント／バックそれぞれ）
   - 選択肢(FE): ローカル状態中心 / グローバルストア（Redux等）/ サーバ状態ライブラリ（TanStack Query 等）/ URL/サーバ駆動
   - 推奨デフォルト: サーバ状態はキャッシュライブラリ、UI状態はローカル、真にグローバルなものだけストア

5. **同期 vs 非同期 / 結合度** — 処理連携はどう設計するか？
   - 選択肢: 同期呼び出し中心 / イベント駆動・非同期 / ジョブキュー併用
   - 推奨デフォルト: 基本は同期、重い処理・通知系のみ非同期化

6. **エラーハンドリングの設計方針** — 失敗をどう扱うか？
   - 選択肢: 例外を投げて境界で集約 / Result型で値として扱う / 併用（ドメインはResult、境界で例外）
   - 推奨デフォルト: ドメインの想定内失敗は Result/型、想定外は例外。握りつぶさない（CLAUDE.md準拠）
   - 深掘り: ユーザー向けエラーメッセージの言語・粒度は？（本プロジェクトは日本語）

7. **拡張性 vs シンプルさ** — 迷ったときどちらを優先するか？
   - 選択肢: 今のシンプルさ優先（YAGNI）/ 将来の拡張余地優先 / ケースバイケース（判断基準を明記）
   - 推奨デフォルト: シンプルさ優先。抽象化は2回目の重複が出てから

8. **非機能の優先順位** — セキュリティ・性能・開発速度・保守性のうち何を最優先するか？
   - 推奨デフォルト: セキュリティと保守性を土台に、性能は計測してから最適化
   - 深掘り: 守るべき性能予算（応答時間・Core Web Vitals 等）はあるか？

9. **技術スタックの制約** — 採用/非採用が決まっている技術は？
   - 確認: 言語・フレームワーク・DB・インフラ・既存資産との整合

## B. 実装思想（CODING_PHILOSOPHY）

コードをどう書くか。

1. **命名規則** — 言語標準に従うか、独自規約があるか？
   - 推奨デフォルト: 言語コミュニティ標準（JS/TS は camelCase、コンポーネントは PascalCase）。略語は避け意図を表す名前
   - 深掘り: ブール・イベントハンドラ・非同期関数の命名規則は？

2. **関数・コンポーネントの粒度** — どのくらいで分割するか？
   - 選択肢: 小さく単一責務に徹底分割 / ある程度のまとまり優先 / 行数より凝集度で判断
   - 推奨デフォルト: 1関数1責務、画面に出る前に長くなったら抽出。凝集度基準

3. **純粋関数・副作用の扱い** — どこまで純粋関数志向にするか？
   - 推奨デフォルト: ロジックは純粋関数に寄せ、副作用（IO・状態変更）は境界に集約

4. **型の厳密さ**（TypeScript 等）— どこまで厳しくするか？
   - 選択肢: strict 全部・any 禁止 / 実用重視で部分的に緩める / 段階的に厳格化
   - 推奨デフォルト: strict 有効・`any` 原則禁止（必要時は理由コメント付き unknown+絞り込み）

5. **コメント方針** — 何を書き、何を書かないか？
   - 推奨デフォルト: 「何を」ではなく「なぜ」を書く。自明なコメントは付けない。公開APIにはドキュメントコメント
   - 深掘り: コメントの言語は？（本プロジェクトは日本語可）

6. **早期 return / ネスト** — ガード節を使うか？
   - 推奨デフォルト: 早期 return でネストを浅く。else を避ける

7. **DRY vs WET** — 重複をどこまで許容するか？
   - 選択肢: 徹底的に共通化 / 早すぎる抽象化を避け重複を一定許容（AHA/WET）
   - 推奨デフォルト: 3回目の重複で共通化を検討。偶然の一致は無理に共通化しない

8. **抽象化のタイミング** — いつ抽象を導入するか？
   - 推奨デフォルト: 具体を2つ以上見てから抽象化。憶測の汎用化はしない

9. **ライブラリ依存方針** — 自作 vs 依存追加の基準は？
   - 選択肢: 積極的に枯れたライブラリを使う / 依存は最小限・小物は自作 / ケースバイケース
   - 推奨デフォルト: 広く使われ保守された物は採用、些末な処理に重い依存を足さない。ライセンス確認

10. **フォーマッタ・リンタ** — 何を使い、CIで強制するか？
    - 推奨デフォルト: Prettier + ESLint（または言語標準）を導入し、整形は自動・差分は機械任せ

11. **シークレット・設定の扱い** — 機微情報をどう扱うか？
    - 必須: APIキーはコードに直接書かない、`.env` はコミットしない・読まない（CLAUDE.md準拠）

## C. テスト思想（TEST_PHILOSOPHY）

品質をどう担保するか。

1. **テストピラミッドの比率** — ユニット／統合／E2E の重みは？
   - 選択肢: ユニット厚め（古典ピラミッド）/ 統合重視（テスティングトロフィー）/ E2E中心
   - 推奨デフォルト: 統合テスト重視（テスティングトロフィー）。ユニットはロジック、E2Eは主要動線のみ

2. **TDD の採用度** — テストを先に書くか？
   - 選択肢: 厳格に TDD / バグ修正時は必ず先に失敗するテスト / 後追いでよい
   - 推奨デフォルト: バグ修正は再現テストを先に（Prove-It）。新規は重要ロジックで TDD

3. **カバレッジ目標** — 数値目標を置くか？
   - 推奨デフォルト: 数値より「重要パスを必ず覆う」を優先。あえて置くなら行 70〜80% 目安、100%は求めない
   - 深掘り: カバレッジ未達で CI を落とすか？

4. **モック方針** — 何をモックし、何を本物で動かすか？
   - 選択肢: 外部I/Oのみモック / 依存は広くモック / なるべく本物（実DB・コンテナ）
   - 推奨デフォルト: 自分の境界の外（ネットワーク・外部API）だけモック。内部はなるべく本物

5. **E2E の範囲** — どこまで自動E2Eを書くか？
   - 推奨デフォルト: クリティカルなユーザー動線（登録・ログイン・決済等）に限定

6. **テストの命名・構成** — 命名規則は？
   - 推奨デフォルト: 「状況→操作→期待結果」が読み取れる記述的な名前。AAA（Arrange-Act-Assert）構成

7. **スナップショットテスト** — 使うか？
   - 推奨デフォルト: 多用しない。壊れやすく意図が薄いものは避け、振る舞いの検証を優先

8. **CI ゲート** — マージ前に何を必須にするか？
   - 推奨デフォルト: lint + 型チェック + テスト + ビルドをパス必須。失敗はマージ不可

9. **フレーキー（不安定）テスト対応** — どう扱うか？
   - 推奨デフォルト: 安易な retry で隠さず原因を直す。直せない間は明示的に隔離し記録（握りつぶさない）

10. **受け入れ条件との対応** — REQUIREMENTS の受け入れ条件をどうテストに紐づけるか？
    - 推奨デフォルト: 各受け入れ条件に対応するテストを最低1つ用意し、対応を追跡可能にする

## D. Git運用方針（GIT_CONVENTIONS）

バージョン管理の運用ルール。出力は `docs/GIT_CONVENTIONS.md`（思想ファイルとは別）。
まず「①Git管理の有無」を聞き、使う場合のみ「②コミット粒度」「③コミットメッセージ規約」を深掘りする。

1. **Git管理の有無** — このプロジェクトで Git を使うか？
   - 推奨デフォルト: 使う
   - 「使わない」場合: その理由を記録し、以降のGit質問はスキップ（GIT_CONVENTIONS.md に判断と理由だけ残す）

2. **ブランチ運用**（使う場合）— どのブランチ戦略か？
   - 選択肢: トランクベース（短命feature・main常時デプロイ可能）/ GitHub Flow / Git Flow
   - 推奨デフォルト: **1機能=1ブランチ=1PR**。`docs/TASKS.md` のタスク単位で `main` から短命feature
     ブランチ（`feature/<ID>-<slug>`）を切り、`main` は常にグリーン（動く状態）に保つ。`main` への
     直接コミットは禁止。機能ごとに `main` への PR を必須とし、**マージは人間が承認して行う**
     （AIはマージしない）。マージ後に次タスクへ。長命ブランチを避ける
   - 深掘り: ブランチ命名規約、`main` 保護ルール、PR本文に含める項目（受け入れ条件・レビュー/テスト結果）

3. **コミット粒度** — 1コミットの大きさの基準は？
   - 選択肢: 1コミット1論理変更 / 機能単位 / 作業セッション単位
   - 推奨デフォルト: 1コミット1論理変更（レビュー・revert 可能な最小単位）
   - 深掘り: WIPコミットの可否、マージ前 squash の方針、整形のみの変更を機能変更と分けるか

4. **コミットメッセージの形式** — どの規約に従うか？
   - 選択肢: Conventional Commits（`type(scope): 件名`）/ 独自プレフィックス / 自由形式
   - 推奨デフォルト: Conventional Commits
   - type 例: feat / fix / docs / refactor / test / chore / perf / style / ci

5. **メッセージの言語** — 日本語か英語か？
   - 推奨デフォルト: 日本語可（CLAUDE.md の言語ポリシー準拠）。プロジェクトで統一する

6. **件名・本文のルール** — 何を書くか？
   - 推奨デフォルト: 件名は簡潔（目安50字）・体言止めまたは命令形で統一・末尾ピリオドなし。
     本文には「なぜ」を書く。Issue/PR 番号を参照する
   - 深掘り: フッターの運用（例: `Co-Authored-By` トレーラー、`BREAKING CHANGE:`）

7. **禁止事項** — コミットしてはいけないものは？
   - 必須: シークレット・APIキー・`.env` をコミットしない（CLAUDE.md準拠）。
     `.gitignore` で生成物・秘匿情報を除外する

8. **タグ・リリース**（任意・必要なら）— バージョニング方針は？
   - 推奨デフォルト: 必要になったら SemVer。当面は不要なら「未定」と記録
