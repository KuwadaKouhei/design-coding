# テスト（Testing）

## Overview

`docs/philosophy/TEST_PHILOSOPHY.md`（品質をどう担保するか）に準拠してテストを設計・実装する。
テストは「正しく動くことの証明」であり、思想で定めた比率・モック方針・カバレッジ方針に沿って書く。
書いたコードの数だけテストがある状態を、思想の範囲で目指す。

## When to Use

- 実装が一段落し、テストを書くとき
- バグを修正するとき（まず失敗する再現テストを書く＝Prove-It）
- 既存機能の振る舞いを保証したいとき

**前提**: `docs/philosophy/TEST_PHILOSOPHY.md` が存在すること。なければ先に `philosophy-definition` を実行。

## Process

### 1. 着手前にテスト思想を読む（必須）

- `docs/philosophy/TEST_PHILOSOPHY.md` — ピラミッド比率・TDD採用度・カバレッジ目標・モック方針・
  E2E範囲・命名・CIゲート・フレーキー対応
- `docs/REQUIREMENTS.md` の受け入れ条件 — 各条件に対応するテストを用意するため
- `docs/DESIGN.md` — テスト対象の境界・IF

### 2. テスト戦略を立てる

TEST_PHILOSOPHY の比率に沿って、何をどのレベル（ユニット／統合／E2E）でテストするか割り当てる。
- ロジック・分岐 → ユニット
- モジュール結合・API → 統合
- クリティカルなユーザー動線 → E2E（思想で定めた範囲に限定）

### 3. 受け入れ条件を網羅する

REQUIREMENTS の各受け入れ条件に、最低1つのテストを対応させる。対応表を作ると追跡しやすい。

```
受け入れ条件「誤った資格情報では『メールまたはパスワードが違います』を表示」
  → 統合テスト: POST /login に誤った資格情報 → 401 と該当メッセージを検証
```

### 4. テストを実装する

TEST_PHILOSOPHY の方針に沿って書く:
- **命名**: 「状況→操作→期待結果」が読み取れる記述的な名前、AAA（Arrange-Act-Assert）構成
- **モック**: 思想で定めた境界だけモック（例: 外部I/Oのみ）。内部はなるべく本物
- **バグ修正**: 先に失敗するテストを書き、修正後にそれが通ることを確認（Prove-It Pattern）
- **スナップショット**: 思想の方針に従う（多用しない等）

### 5. 実行・確認する

- テストを実行し、全件パスを確認する
- カバレッジを測り、思想の目標（重要パス網羅・数値目安）に達しているか確認する
- フレーキーなテストは安易な retry で隠さず原因を直す。直せない間は明示的に隔離し記録する（握りつぶさない）

### 6. CIゲートを反映する

TEST_PHILOSOPHY で定めたゲート（lint + 型 + テスト + ビルド等）がマージ前に必須になるようにする。
設定が未整備なら、整備をユーザーに提案する。

### 7. 変更点を説明する

追加・変更したテストと、その狙いを日本語で説明する（CLAUDE.md準拠）。

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「動いているからテストは後でいい」 | 「後で」はほぼ来ない。テストの価値は回帰を即検知すること。今書く。 |
| 「テスト思想を読まなくても書ける」 | 比率・モック方針が思想と食い違うと、過剰／過小テストになりレビューで揉める。 |
| 「実装に合わせてテストを書けばいい」 | 実装内部に密結合したテストは壊れやすい。振る舞い（受け入れ条件）をテストする。 |
| 「バグ修正にテストは要らない」 | 再現テストが無いと再発を防げない。まず失敗するテストを書くのが原則（Prove-It）。 |
| 「フレーキーはretryで通せばいい」 | retry は不具合を隠す。CLAUDE.md の握りつぶし禁止に反する。原因を直す。 |
| 「カバレッジ100%にすべき」 | 数値の盲目的追求は無意味なテストを生む。重要パスの網羅を優先する。 |

## Red Flags

- TEST_PHILOSOPHY を読まずにテストを書き始めている
- 受け入れ条件に対応しないテストばかり、または対応するテストが無い受け入れ条件がある
- 思想で「外部I/Oのみモック」なのに内部実装まで広くモックしている
- バグ修正に再現テストが付いていない
- フレーキーなテストを retry や skip で隠している
- カバレッジ数値だけ追って意味の薄いテストを量産している

## Verification

完了とみなす前に確認する:

- [ ] 着手前に TEST_PHILOSOPHY.md を読んだ
- [ ] テストの構成（ユニット/統合/E2E）が思想の比率に沿っている
- [ ] すべての受け入れ条件に対応するテストがある
- [ ] モック範囲が思想どおり
- [ ] バグ修正には失敗→修正で通る再現テストがある
- [ ] 全テストがパスし、カバレッジ目標を満たしている
- [ ] フレーキーなテストを隠していない
- [ ] CIゲート（lint/型/テスト/ビルド）が反映されている
- [ ] 追加・変更したテストと狙いを日本語で説明した

## フェーズ完了

要件定義 → 設計（思想定義含む）→ 実装 → テスト の一巡が完了する。要件・思想・設計のドキュメントは
生きた文書として、変更が生じたら更新する。
