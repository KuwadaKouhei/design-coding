# ディレクトリ構造設計（Directory Structure）

## Overview

`docs/DESIGN.md`（論理的なコンポーネント分割）・`docs/TECH_STACK.md`（採用FWの規約）・
`docs/DATABASE.md`（あれば）・`docs/philosophy/PLAN_PHILOSOPHY.md`（境界・依存方向）を入力に、
**実際のフォルダ／ファイル階層**を確定し `docs/DIRECTORY_STRUCTURE.md` として書き出す。

ディレクトリ構造は**常に必ず可読性と拡張性を最優先**で設計する。

- **可読性**: どこに何があるかが名前から分かる。1ディレクトリ1責務。深すぎない階層。迷わず目的のファイルに辿り着ける。
- **拡張性**: 新しい機能・要素を足すとき「どこに何を追加するか」が一意に決まる。依存方向が壊れない。
  `shared`/`common`/`utils` が無秩序に肥大化しない。

思想に反する判断をするときは、その理由を明示する。

## When to Use

- アーキテクチャ設計（`docs/DESIGN.md`）が完了したあと（必要なら DB設計 `docs/DATABASE.md` の後）
- `/3-claude-md`（CLAUDE.md生成）・実装に着手する前
- 「ファイルをどこに置くか」のルールをチーム／将来の自分のために確定したいとき

**前提**: `docs/DESIGN.md` が存在すること。なければ先に `design-and-architecture` を実行。

## Process

### 1. 入力を読む

- `docs/DESIGN.md` — コンポーネント／モジュール構成と責務・依存関係
- `docs/TECH_STACK.md` — 採用フレームワーク（FW固有のディレクトリ規約に従う／逸脱するかを決める前提）
- `docs/DATABASE.md`（あれば） — モデル／マイグレーションの置き場所に影響
- `docs/philosophy/PLAN_PHILOSOPHY.md` — 境界・依存方向・優先順位
- 既存コードがあれば実際のディレクトリ構造を確認し、再利用できる構造・規約を踏襲する

### 2. 構造方針（基調）を決める

レイヤー型 / 機能（feature）型 / ドメイン型 のどれを基調にするかを、PLAN_PHILOSOPHY と DESIGN の
分割方針に合わせて決める。採用FWの標準レイアウト（例: Next.js の `app/`、Rails の規約）と整合させ、
逆らう場合は理由を残す。

### 3. 可読性のルールを定義する

- **命名規約**: ケース（kebab/camel/Pascal/snake）、複数形/単数形、接尾辞（`.service`/`.test` 等）
- **1ディレクトリ1責務**: ディレクトリ名から中身の役割が分かること
- **階層の深さ**: 深くしすぎない目安を決める（例: 原則3〜4階層まで）
- **同居 or 分離**: テスト・型・スタイルをコードと同居させるか分けるか
- **index/barrel** の方針（再エクスポートを許すか）

### 4. 拡張性のルールを定義する

- **追加先が一意に決まる配置**: 「新しい機能／エンドポイント／モデルを足すときどこに置くか」を明文化
- **依存方向**: PLAN_PHILOSOPHY に沿った依存の向き（例: 内側＝ドメインは外側＝詳細に依存しない）
- **境界をまたぐ import 規約**: どのディレクトリから何を import してよいか
- **共有コードの抑制**: `shared`/`common`/`utils` の肥大化を防ぐ基準（昇格条件・分割条件）

### 5. ツリー図と責務表を書く

- 実フォルダ構成を**コードブロックのツリー**で示す
- 各トップレベルディレクトリの**責務表**（ディレクトリ / 責務 / 置くもの・置かないもの）を書く

### 6. 「追加手順の例」で自己検証する

「新しい〇〇（機能・画面・モデル等）を追加する」例を1〜2件、どのファイルをどこに足すか具体的に示し、
ルールが実際に機能し迷いが生じないことを確認する。

### 7. 思想との整合を確認し、逸脱を記録する

各配置判断が PLAN_PHILOSOPHY・TECH_STACK の規約に沿っているか自己点検する。やむを得ず逸脱する場合は、
理由を明記する（黙って逸脱しない）。

### 8. docs/DIRECTORY_STRUCTURE.md に保存し、人間のレビューを受ける

```markdown
# ディレクトリ構造: <プロジェクト名>

## 1. 概要 / 構造方針（PLAN_PHILOSOPHY・TECH_STACKとの対応）
## 2. ディレクトリツリー
## 3. 各ディレクトリの責務
## 4. 命名規約（可読性）
## 5. 依存方向・配置ルール（拡張性）
## 6. 拡張手順の例（新しい〇〇を足すとき）
## 7. 設計思想からの逸脱（あれば、理由付き）
## 8. 未決事項
```

## Common Rationalizations

| 言い訳 | 実際 |
|---|---|
| 「フレームワークのデフォルトのままで良い」 | デフォルトは出発点にすぎない。プロジェクト固有の配置を言語化しないと、人によって置き場所がぶれて拡張時に破綻する。 |
| 「構造は書きながら自然に決まる」 | 後から構造を直すのは大量のファイル移動と import 修正を伴い高コスト。先にルールを固める価値が大きい。 |
| 「util に入れておけば後で整理する」 | `util`/`common` は整理されず肥大化する。昇格・分割の基準を先に決めなければ無秩序になる。 |
| 「ツリーを貼れば十分」 | ツリーだけでは「次に何をどこへ足すか」が分からない。命名・依存・配置ルールこそが拡張性を担保する。 |
| 「思想から外れるが些細だから黙ってやる」 | 黙った逸脱は構造の一貫性を壊す。明記すれば合意のうえの例外になる。 |

## Red Flags

- `docs/DESIGN.md` を読まずに構造を決めている
- 責務が曖昧な `util`/`common`/`misc` ディレクトリが乱立している
- 階層が深すぎて目的のファイルに辿り着けない
- 「新しい機能をどこに置くか」のルールが定義されていない
- 依存方向の制約が無く、どこからでも何でも import できる
- 命名規約が場所によってバラバラ
- 思想・FW規約に反しているのに理由が書かれていない

## Verification

実装フェーズ（CLAUDE.md生成）に進む前に確認する:

- [ ] DESIGN.md・TECH_STACK.md（と必要なら DATABASE.md）・PLAN_PHILOSOPHY を入力にしている
- [ ] 構造の基調（レイヤー/機能/ドメイン）と FW規約との整合が決まっている
- [ ] **可読性**のルール（命名・1責務・深さ・同居/分離）が定義されている
- [ ] **拡張性**のルール（追加先の一意性・依存方向・共有の抑制）が定義されている
- [ ] ディレクトリツリーと各ディレクトリの責務表がそろっている
- [ ] 「新しい〇〇を足す」例で配置に迷いが生じないことを確認した
- [ ] 思想・規約からの逸脱があれば理由付きで明記されている
- [ ] 人間がレビューし承認した
- [ ] `docs/DIRECTORY_STRUCTURE.md` に保存されている

## 次のステップ

ディレクトリ構造が確定したら設計フェーズは完了。`/3-claude-md` で CLAUDE.md を生成し、`/4-implement`
（`implementation` スキル）で実装へ進む。実装は `docs/DIRECTORY_STRUCTURE.md` の配置規約・命名規約・
依存方向に準拠して進める。構造を変えるときは本ドキュメントも一緒に更新する。
